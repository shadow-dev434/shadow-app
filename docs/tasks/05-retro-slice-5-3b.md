## Pattern operativo emerso

La sessione del 2026-04-29 ha esercitato un modo di lavorare a tre voci che merita di essere descritto come meccanica osservabile, non come virtù collettiva. Tre attori: Antonio (autore), Claude assistant (questa istanza, in chat web), Claude Code (CLI nel repo locale). Ognuno con un ruolo strutturalmente distinto, e — questa è la cosa che vale davvero — con responsabilità di **controllo reciproco** definite in modo asimmetrico. Il pattern non è "tre teste meglio di una", è "tre teste in cui ognuna controlla un'area specifica delle altre due".

### La divisione effettiva dei tre ruoli

**Antonio** prende le decisioni cardinali (le 4 documentate sopra più decine di micro-decisioni di setup) e gestisce manualmente l'esecuzione browser-based durante i turni E2E (apertura della chat in dev, immissione dei messaggi turno-per-turno, copia-incolla degli output verbatim al termine di ogni turno). Non scrive codice durante la sessione E2E — la disciplina "niente patch in-flight" lo vincola a fare il decisore strategico, non l'implementatore. Mantiene la rubric pre-registrata e applica la valutazione PASS/FAIL meccanicamente.

**Claude assistant** lavora in chat web con accesso ai file di progetto (`docs/tasks/05-*.md`, `schema.prisma`, `ROADMAP.md`) caricati nel context. Funzioni esercitate durante la sessione: second opinion strategica sulle proposte di Claude Code, formulazione delle opzioni decisionali in punti di biforcazione (le 4 decisioni sono state presentate ad Antonio come liste numerate di alternative con costo/beneficio per ognuna), scrittura dei briefing operativi per Claude Code in formato copy-paste-pronto, sintesi degli output verbatim di Claude Code in verdetti rubric. Non esegue codice né tocca filesystem — è puro layer di pianificazione e sintesi.

**Claude Code** lavora nella CLI dentro `C:\shadow-app` con accesso filesystem completo, esegue tool call (Bash, Read, Write, Edit), legge schema/codice/DB direttamente, esegue script di setup (`temp-shift-profile-style.ts`, `seed-e2e-s1.ts`), scrive diff e li applica solo dopo approvazione esplicita di Antonio. Funzione strutturalmente distintiva: **verifica contro la fonte primaria**. Quando Claude assistant fa affermazioni su scope, file:linea, schema Prisma, contenuto del commit body, Claude Code le verifica leggendo il file effettivo prima di eseguire — non le esegue ciecamente.

### Le due interruzioni di scope (il pattern più importante)

Durante la sessione, Claude assistant ha commesso due errori di lettura che, se eseguiti senza verifica, avrebbero compromesso la qualità diagnostica della sessione. Entrambi sono stati intercettati e corretti da Claude Code.

**Primo errore — scope di Scenario 5.** Claude assistant aveva scritto un briefing per S5 assumendo che lo scenario testasse l'asse 3.3 della spec (frizione emotiva, "uffa lasciamo perdere"). Claude Code ha verificato il commit body letteralmente prima di eseguire il setup, ha rilevato il mismatch, ha fermato l'esecuzione e ha rimandato la correzione ad Antonio.

**Secondo errore — premessa sul fail D4.** Dopo la diagnosi del fail D4 in S4, Claude assistant aveva inizialmente raccomandato Opzione 1 (stop sessione) sull'assunto implicito che "il fail D4 non si ripeterà in S5 perché S5 non ha lo stesso trigger di decomposizione". Le due interruzioni hanno una caratteristica strutturale comune: Claude Code non si è limitato a eseguire il task assegnato, ha **verificato la coerenza tra il task assegnato e la fonte primaria** (commit body, schema, codice). Quando il task era incoerente con la fonte, ha fermato l'esecuzione e ha sollevato il dubbio. Questa è la regola operativa centrale di questa sezione.

### Regole operative replicabili

Le seguenti regole sono ancorate a episodi specifici della sessione e replicabili in future E2E.

**(R1) Claude Code verifica le affermazioni di Claude assistant contro la fonte primaria, non le esegue ciecamente.** Ancoraggio: le due interruzioni di scope appena descritte. Quando Claude assistant scrive nel briefing operativo "S5 testa frizione emotiva" o "linee 124-160 del prompt", Claude Code apre il commit body / il file effettivo prima di eseguire e segnala il mismatch se c'è. Questa regola ha valore solo se è **bidirezionale e non gerarchica**: Claude Code non deve "fidarsi" di Claude assistant, deve "verificarlo come verifica qualsiasi altra fonte secondaria". Asimmetria di accesso (Claude Code legge i file effettivi, Claude assistant lavora con file caricati che possono essere stale o parziali) rende questa verifica strutturalmente necessaria.

**(R2) I briefing operativi passano via copy-paste esplicito, non via delega vaga.** Ancoraggio: durante la sessione, oltre 15 messaggi del tipo "Copia e incolla per Claude Code: ---" con il blocco completo che Antonio doveva trasferire dalla chat web alla CLI. Il vantaggio non è solo precisione — è che il briefing diventa **artefatto verificabile**: Antonio rilegge prima di incollare, può fermare un briefing sbagliato prima dell'esecuzione, può citare il briefing testualmente in caso di disallineamento downstream. La delega vaga ("dì a Claude Code di fare il setup di S5") perde questi punti di controllo.

**(R3) Lookup pre-turni come check di stato verificabile, non come fiducia nello stato implicito.** Ancoraggio: prima di ogni scenario E2E, lookup dedicato (`bunx dotenv-cli ... bun -e "..."`) che stampa task non-terminali, profilo, active thread, review della giornata. Output verbatim copiato in chat. Permette ad Antonio di verificare PRE-browser che il setup è quello atteso (3 task non-terminali, profilo=gentle, active thread=null), e fa scattare archive thread sticky se ne emerge uno (8 episodi sticky di giornata, tech debt #1 ricorrente). Senza lookup, lo stato implicito può essere inconsistente con quello atteso e i fail rubric diventano ambigui ("è bug del prompt o setup sporco?").

**(R4) Disciplina "niente patch in-flight" protegge l'integrità della rubric pre-registrata.** Ancoraggio: la tentazione naturale di patchare il prompt evening_review tra S4 e S5 (per provare a far passare D4) è stata respinta. Patchare in-flight contamina la sessione: i fail successivi sarebbero ambigui ("bug originale o regressione del fix?"). La regola si combina con (R3): lookup pre-turni verifica che il setup sia coerente, "niente patch" verifica che il sistema sotto test sia immutato. Insieme rendono la rubric pre-registrata applicabile meccanicamente.

**(R5) "Qual è il valore informativo qui?" come domanda decisionale ai punti di biforcazione, non "come chiudo prima?".** Ancoraggio: le 4 decisioni cardinali. La domanda "valore informativo" produce le 4 decisioni controintuitive che hanno trasformato la sessione da "5 scenari falliti, promozione bloccata" a "5 scenari falliti, promozione bloccata, 15 tech debt strutturati con root cause + fix path V1.1 specifico". La domanda "come chiudo prima?" avrebbe prodotto solo i numeri della rubric.

**(R6) Le decisioni cardinali sono di Antonio, non di Claude.** Ancoraggio: nelle 4 decisioni, Claude assistant ha sempre formulato le opzioni e indicato la sua inclinazione, ma la scelta finale è stata di Antonio. In due casi (D2 sovrascrittura profilo, D3 Opzione C) Antonio ha scelto un'opzione diversa da quella che Claude assistant aveva inizialmente suggerito ("Opzione 1 stop totale" per D1; "B skip S5" come baseline per D3). La regola implica che il pattern non è "Claude propone, Antonio approva" — è "Claude formula opzioni e costo/beneficio, Antonio decide". Una sfumatura non triviale: la prima formulazione invita Antonio ad approvare la migliore proposta di Claude, la seconda lo invita a scegliere tra opzioni equivalenti dal punto di vista informativo presentate da Claude. La seconda è la vera divisione del lavoro.

### Precondizioni che hanno reso il pattern efficace

Il pattern non funziona in assenza di tre precondizioni strutturali, tutte presenti in questa sessione e tutte verificabili pre-sessione.

**Pre-registrazione formale rigorosa.** I 5 scenari erano pre-registrati (in sessione precedente) con rubric binaria, vincoli di promozione binari (5/5 comportamentale, 4/5 testuale con condizioni, ordering 5/5), scope esplicito per ogni scenario derivato dal commit body. Senza questa pre-registrazione, la rubric meccanica non ha àncora — ogni fail diventa interpretabile, e l'interpretazione live a fine giornata è la cosa che produce falsi PASS.

**Disciplina sui ruoli mantenuta nei momenti di stanchezza.** A 6+ ore di sessione la tentazione naturale è la convergenza dei ruoli ("Antonio guarda il diff e decide insieme a Claude Code", "Claude assistant esegue uno script al posto di Antonio"). I tre ruoli sono rimasti separati anche a fine sessione — Antonio non ha mai modificato direttamente file, Claude Code non ha mai preso decisioni cardinali, Claude assistant non ha mai eseguito tool call al posto di Claude Code. La separazione è quello che ha reso possibile (R1) — la verifica reciproca funziona solo tra ruoli distinti.

**Capacità di Antonio di sospendere lo script.** Le 4 decisioni cardinali esistono solo perché Antonio ha permesso alla sessione di **uscire dal piano pre-registrato** in punti specifici. Una sessione "rispetta lo script o niente" avrebbe prodotto i 5 scenari completati senza investigazione D4, senza Opzione C, senza ipotesi sovrascrittura profilo. La capacità di sospendere lo script è strutturalmente diversa dalla "flessibilità generica" — è la capacità di riconoscere **i punti specifici** in cui valore informativo emergente eccede valore informativo del piano pre-registrato. La differenza tra le due è cosa rende il pattern non collassabile in "ognuno fa quello che gli pare se sembra utile".

## Lezioni metodologiche

Le lezioni che seguono sono distinte dalle decisioni cardinali (eventi singoli) e dalle regole operative (meccanica replicabile). Sono **tensioni metodologiche generali** che la sessione ha messo a fuoco — pattern che chi farà la prossima E2E incontrerà di nuovo, con scope diverso ma forma simile. Quattro lezioni, ognuna ancorata a un episodio specifico ma più ampia di esso.

### L1 — Rubric meccanica e investigazione live sono complementari, non in conflitto

La disciplina E2E pre-registrata di questa sessione era stringente: "niente patch in-flight, scenario fail con bug noto = annota e prosegui, applicazione meccanica della rubric pre-registrata". Letta letteralmente, questa disciplina sembra incompatibile con quello che è successo dopo Scenario 4: stop, lettura prompt + orchestrator, diagnosi root cause prima di Scenario 5. Investigazione live è esattamente quello che la disciplina pre-registrata sembrerebbe vietare.

In realtà non c'è contraddizione, ma una divisione del lavoro tra due strumenti complementari. La rubric meccanica produce **dati puliti**: S4 D4 fail, S2/S5 testuale fail, ordering 5/5 PASS. Questi dati hanno valore probatorio solo se la rubric viene applicata in modo non interpretativo, e questo richiede di mantenere il sistema sotto test immutato (niente patch) e la rubric immutata (criteri PASS/FAIL pre-registrati). L'investigazione live produce **diagnosi pulite**: root cause specifica del fail D4, distinzione tra ipotesi (a)/(b)/(c), fix path V1.1 articolato in due strade. La diagnosi richiede uscita dallo script — leggere codice non testato, formulare ipotesi non pre-registrate, accettare zone di ignoranza.

I due strumenti hanno output diversi e regimi di applicazione diversi. La rubric meccanica è applicabile *durante* gli scenari E2E, l'investigazione live è applicabile *tra* gli scenari (o dopo l'ultimo). Confonderli — applicare interpretazione live durante lo scenario, oppure mantenere rubric meccanica anche per la diagnosi del fail — collassa il valore di entrambi. Una rubric "interpretata live" produce verdetti negoziabili a fine giornata. Una diagnosi "meccanica" produce fix path generici tipo "hardening prompt" che non discriminano tra cause diverse.

La regola operativa che ne deriva: pre-registrare la rubric meccanica per la valutazione binaria del sistema sotto test, riservare margine esplicito per investigazione live tra scenari quando il fail è informativo (vedi L4 sotto). Le due cose non sono "scelta tra disciplina e flessibilità", sono **due fasi sequenziali della stessa sessione**.

### L2 — Un'ipotesi smentita produce informazione, se è formulata bene

L'ipotesi "sovrascrittura parziale del profilo adattivo invalida il fail testuale gentle di S2/S5" è stata smentita dall'investigazione: `buildUserContext` espone solo 4 campi al modello, di cui 1 solo discriminante (`preferredPromptStyle`); il prompt evening_review legge solo `preferredPromptStyle`; sovrascrivere quel campo è semanticamente sufficiente. Ipotesi falsa, verdetto Slice 5 invariato.

Però la smentita ha prodotto il tech debt #15: mismatch tra dimensioni dichiarate da `05-review-serale-spec.md` per V1 (`preferredPromptStyle`, `preferredTaskStyle`, `shameFrustrationSensitivity`) e dimensioni effettivamente esposte in `buildUserContext` (solo `preferredPromptStyle` discriminante, le altre tre esposte sono baseline). Lo style "gentle" arriva al modello come etichetta cosmetica senza contesto rinforzante. Implicazione concreta per V1.1: Strada 1 prompt hardening da sola potrebbe non bastare, gli esempi few-shot devono sovrastare l'inerzia "modello produce direct di default" senza pressure stilistica nel modeContext.

Il pattern generalizzabile: il valore di un'ipotesi non è binario "confermata = utile / smentita = inutile". Un'ipotesi ben formulata produce **un quadro investigativo** che vale indipendentemente dall'esito. La verifica della sovrascrittura profilo richiedeva di leggere `buildUserContext` riga per riga, leggere `EVENING_REVIEW_PROMPT` cercando cross-reference ad altre dimensioni, ispezionare il profilo del test user campo per campo. Quel lavoro è il sottoprodotto: anche se l'ipotesi cade, il quadro che l'investigazione ha rivelato resta — e in questo caso ha mostrato un debito architetturale latente che senza l'ipotesi sarebbe rimasto invisibile fino a tester reali.

La regola operativa: formulare le ipotesi in modo che la smentita sia altrettanto informativa della conferma. Concretamente, prima di lanciare l'investigazione, chiedere "se questa ipotesi è smentita, cosa avrò scoperto comunque?". Se la risposta è "niente, torno a dove ero prima", l'investigazione non vale il costo. Se la risposta è "avrò mappato il segmento X dell'architettura / verificato il flag Y / chiarito la semantica Z", l'investigazione vale a prescindere. Nel caso D2, la risposta era "avrò mappato cosa il prompt evening_review effettivamente legge dal profilo adattivo" — informazione che vale per qualsiasi futura modifica del prompt o del profilo, indipendentemente dall'esito specifico dell'ipotesi.

### L3 — La fatica produce deriva specifica, non generica

A 6+ ore di sessione la concentrazione cala — banale. Quello che è meno banale è che la deriva ha **forma specifica**, riconoscibile, e non è "errore casuale". Tre sintomi che ho osservato emergere durante questa sessione, contrastati prima che producessero danno.

**Sintomo 1: convergenza dei ruoli.** A fine sessione, la tentazione naturale è la fusione dei tre attori — Antonio comincia a guardare il diff insieme a Claude Code per "decidere insieme", Claude assistant comincia a suggerire script da eseguire al posto di formulare opzioni, Claude Code comincia ad accettare briefing vaghi senza richiedere copy-paste esplicito. La separazione dei ruoli (precondizione "disciplina sui ruoli mantenuta") è uno degli ancoraggi di qualità della sessione, e la fatica la attacca per prima perché la separazione costa attenzione esplicita.

**Sintomo 2: scivolamento dalla domanda decisionale alla domanda di chiusura.** La domanda operativa "qual è il valore informativo qui?" (R5) richiede di tenere aperto un orizzonte costoso: ogni biforcazione comporta valutare costo/beneficio in termini di informazione prodotta, non in termini di tempo speso. La domanda "come chiudo prima?" è strutturalmente più semplice e produce risposte più rapide. Non è un errore — è un cambio di funzione obiettivo, e a fine giornata è la deriva più subdola perché si presenta come "buon senso pratico". Le 4 decisioni cardinali sono tutte risposte a "valore informativo", e la 4ª (D2 sovrascrittura profilo) è particolarmente significativa perché è arrivata a sessione quasi chiusa, esattamente nel punto di massima pressione di chiusura.

**Sintomo 3: abbassamento della soglia di accettazione delle proposte.** A fine sessione, la tentazione naturale è accettare la prima proposta ragionevole invece di formulare opzioni alternative. Claude assistant durante la sessione ha sempre formulato opzioni (3 alternative per D1, 2 per D2, 3 per D3, 3 per D4) — anche quando una era chiaramente preferibile. La pratica della formulazione esplicita di opzioni alternative è anti-deriva: costringe a esplicitare il costo/beneficio dell'opzione "ovvia" rispetto a quelle scartate, e in due casi (D2 e D3) Antonio ha scelto un'opzione che non era quella che Claude assistant aveva inizialmente suggerito.

La regola operativa: monitorare i sintomi specifici della deriva, non la fatica in sé. La fatica è inevitabile in sessione lunga; la deriva è prevenibile se si riconoscono i sintomi prima che producano danno. Concretamente: se Antonio si accorge di guardare il diff "insieme a Claude Code" invece di approvare-o-rigettare, è sintomo 1. Se le decisioni iniziano a essere prese in 1 messaggio invece di 3-4 (formulazione opzioni → discussione → scelta), è sintomo 3. Se la domanda interna diventa "manca molto?" invece di "cosa stiamo imparando?", è sintomo 2.

### L4 — La pre-registrazione include il criterio di sospensione, non solo lo script

Questa è la lezione più scomoda perché è autocritica strutturale.

I 5 scenari di Slice 5 commit 3b erano pre-registrati in modo rigoroso: rubric binaria (G1∧G3 per gentle, marker challenge per challenge), vincoli di promozione binari (5/5 comportamentale, 4/5 testuale, ordering 5/5), scope esplicito di ogni scenario derivato dal commit body, disciplina E2E ("niente patch in-flight, scenario fail con bug noto = annota e prosegui"). La pre-registrazione era completa per il caso "tutto va come previsto" — ogni scenario esegue, ogni rubric si applica, ogni verdetto si calcola.

Però la pre-registrazione era **parziale** per il caso "qualcosa va storto in modo informativo". Le 4 decisioni cardinali sono state tutte prese **fuori dallo script pre-registrato**: D1 (investigazione D4) era violazione esplicita di "annota e prosegui"; D2 (sovrascrittura profilo) era riapertura del verdetto a sessione quasi chiusa; D3 (Opzione C su S5 turni 1-2) era esecuzione parziale di uno scenario nominalmente sopra-soglia di abbandono; D4 (committare solo 2 dei 3 temp-*) era pulizia conservativa con criterio non pre-formulato. Tutte e quattro decise live, sulla base del giudizio di Antonio in quel momento.

Funzionato bene questa volta, ma non replicabile come regola. Antonio era ancora lucido nei punti di decisione — abbastanza da riconoscere che D2 era un'ipotesi che valeva l'investigazione anche a sessione quasi chiusa. Una sessione futura potrebbe avere fatica peggiore (8-9 ore invece di 7), un'ipotesi più subdola (richiede investigazione di 30 minuti, non 10), un fail comportamentale meno ovvio (richiede di ricostruire 3 turni invece di leggere il payload del turno 3). In quei casi, "decidiamo lì sul momento" non è abbastanza. Il rischio è che una decisione cardinale venga collassata in "chiudiamo come da piano" perché il criterio per sospendere non è pre-registrato e la fatica favorisce la chiusura.

La regola operativa: la pre-registrazione di future sessioni E2E dovrebbe includere **criteri espliciti di sospensione dello script**, almeno per le situazioni più ricorrenti. Esempi formulabili a priori: "se fail comportamentale strict (es. tool call fuori sequenza) prima dell'ultimo scenario, sospendere e decidere tra (a) investigazione live, (b) annotazione e prosegui, (c) stop, sulla base di stima costo investigazione e ripetibilità del fail su scenari rimanenti"; "se a fine sessione emerge ipotesi di artefatto del setup E2E che potrebbe invalidare uno o più verdetti rubric, sospendere chiusura e investigare se costo < 15 minuti, indipendentemente da ora di giornata e stanchezza"; "se uno scenario può essere eseguito in versione ridotta (es. turni 1-2 invece di 1-3) per produrre informazione discriminante senza riprodurre fail noti, valutare versione ridotta come opzione esplicita prima di decidere skip vs completo".

Il punto chiave: pre-registrare i criteri di sospensione **non rende la sessione più rigida**, la rende più disciplinata nei punti di stress. Il giudizio live di Antonio resta indispensabile (i criteri pre-registrati non possono coprire tutti i casi possibili), ma viene applicato dentro un perimetro che la fatica non può erodere. Il piano "rispetta lo script o niente" e il piano "decidiamo lì cosa fare" sono entrambi insufficienti. Il piano replicabile è "lo script include i punti in cui sospendere lo script, e i criteri per decidere come sospendere".

Tre item meritano qui un richiamo esplicito perché sono strettamente legati alle lezioni metodologiche e al pattern operativo.

**Tech debt #1 — Lazy archive generica per thread non-evening.** Otto episodi sticky di giornata (morning_checkin orfano + thread evening_review residui da scenari precedenti) hanno richiesto archive manuale via script ad-hoc per non bloccare il guard di `GET /api/chat/active-thread`. La regola operativa (R3) — "lookup pre-turni come check di stato verificabile" — è in parte un workaround di questa carenza architetturale. **Tech debt #14 — Fail D4 prompt + orchestrator (causa diagnosticata).** Diagnosi della Decisione 1: il prompt evening_review ha esempi della sequenza turno-N-proposta vs turno-(N+1)-conferma vs turno-(N+2)-tool, ma manca il divieto esplicito di chiamare il tool inline al turno della proposta; l'orchestrator esegue `approve_decomposition` senza state-check sul `thread.contextJson` o sull'ultimo `userMessage`. **Tech debt #15 — Mismatch dimensioni AdaptiveProfile dichiarate vs esposte.** Diagnosi della Decisione 2: `buildUserContext` espone al modello 4 campi su ~40 di `AdaptiveProfile`, di cui solo `preferredPromptStyle` discriminante; le altre tre esposte (`averageCompletionRate`, `averageAvoidanceRate`, `activationDifficulty`) sono baseline numerici per nuovi user. La spec `05-review-serale-spec.md` dichiara per V1 tre dimensioni (`preferredPromptStyle` + `preferredTaskStyle` + `shameFrustrationSensitivity`), ma due su tre non sono effettivamente passate al modello. Implicazione per V1.1: Strada 1 prompt hardening da sola potrebbe non bastare, gli esempi few-shot devono sovrastare l'inerzia "modello produce direct di default" senza pressure stilistica nel modeContext. Tre approcci di costo crescente — solo Strada 1, oppure + estensione `buildUserContext`, oppure + nudge linguistico esplicito.

I restanti 12 item (issue di tooling, pattern operativi minori, miglioramenti rubric, edge case di seed) sono tracciati nelle deploy-notes con la stessa struttura.
