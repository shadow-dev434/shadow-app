# W4 — Bilinguismo it/en (next-intl, senza routing per locale)

> Dipende da W1 (`UserProfile.locale`). Long-tail parallelizzabile con tutto.
> Decisione D10. next-intl 4.3.4 è GIÀ in package.json (inerte, zero import).

## 1. Infrastruttura (prima fase, piccola)

- `next.config.ts`: wrap con `createNextIntlPlugin('./src/i18n/request.ts')`
  (⚠️ file protetto → conferma esplicita prima dell'edit).
- `src/i18n/request.ts` (nuovo): `getRequestConfig` → locale da cookie
  `shadow-locale` → header `Accept-Language` (it|en) → default `'it'`; carica
  `messages/${locale}.json`.
- **NIENTE prefisso URL** `/en/...`: l'app è una SPA client dietro auth (no SEO
  multilingua) e il gate middleware tour→consent→onboarding usa path fissi.
- `src/middleware.ts`: nella `findUnique` edge già esistente (riga ~79)
  aggiungere `locale` alla select; se cookie assente/diverso dal profilo,
  settarlo sulla response. Zero query extra. (⚠️ file core → conferma.)
- `src/app/layout.tsx`: `const locale = await getLocale()`, `<html lang={locale}>`,
  `<NextIntlClientProvider>`, `generateMetadata` localizzato.
- Persistenza: `UserProfile.locale` (default = Accept-Language al register);
  toggle lingua in SettingsView → `PATCH /api/profile` + set cookie + `router.refresh()`.

## 2. Dizionari

`messages/it.json` + `messages/en.json`, namespace: `common, nav, auth, tasks`
(con `categories, contexts, quadrants, decisions, modes, timeOptions`), `focus,
review, eisenhower, settings, chat, onboarding, tour, consent, billing, nudges,
decomposition, assistant, errors`. L'italiano corrente è la fonte; `en.json`
tradotto a blocchi. Script `scripts/check-i18n-keys.ts`: chiavi it/en identiche.

## 4. Engine server-side (~270 stringhe) e API

- `src/lib/i18n/server-translator.ts` (nuovo): `getT(locale, namespace)` basato
  su `createTranslator` di next-intl core (usabile fuori dal request context,
  import statico dei due JSON).
- `generateNudgeMessage` (nudge-engine.ts:~125), ai-assistant-engine,
  decomposition-engine: parametro `locale` (default 'it'); i ~150 template
  literal `${context.taskTitle}` diventano messaggi ICU `{taskTitle}` nel
  namespace `nudges`. Shape dei tipi consumati dalla UI invariata.
- Errori API → **codici stabili** (`{ error: 'task_not_found' }`) tradotti
  client-side nel namespace `errors`; campo `message` italiano mantenuto
  temporaneamente per retro-compat.

## 5. Prompt LLM (`src/lib/chat/prompts.ts` ⚠️ file core → conferma)

**Master unico in italiano + direttiva lingua + esempi localizzati** (NO fork
per locale: è il file più iterato del progetto, il fork crea drift garantito).
- `CORE_IDENTITY` → `buildCoreIdentity(locale)`: corpo invariato + blocco
  `LINGUA: rispondi SEMPRE in …` + esempi/Quick Reply per locale.
- Prompt per modalità → funzioni `(locale) => string` SOLO dove contengono
  esempi user-visible; regole/state-machine restano in italiano.
- `buildSystemPromptParts(..., locale)`; `formatDeadlineLabel`
  (`src/lib/evening-review/dates.ts:119`) e `formatPlanPreviewForPrompt`
  (`src/lib/evening-review/plan-preview.ts:203`) parametrizzati per locale —
  file NON protetti, ma `dates.test.ts`/`plan-preview.test.ts` asseriscono le
  stringhe italiane esatte e vanno adeguati.
- **Cache**: locale fissato per-thread al turno 1 (cambio lingua vale dai thread
  successivi) → niente invalidazioni mid-thread. `OrchestratorInput.locale` da cookie.

## Acceptance

1. `<html lang>` riflette il cookie; toggle persiste su `UserProfile.locale` e
   sopravvive a logout/login; fallback Accept-Language→it.
2. Per ogni fase di estrazione: build verde + smoke visivo + zero stringhe
   hardcoded residue (grep euristico su `label:`/lettere accentate).
3. `scripts/check-i18n-keys.ts` verde (chiavi it/en identiche).
4. Thread chat EN end-to-end: risposte, QR, plan preview in inglese; thread IT
   **byte-identici** nello staticPrefix (protezione cache + campagne validate).
5. Il SW non serve HTML stale dopo cambio lingua (le navigation lo bypassano già).
